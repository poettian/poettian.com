server_name的作用

当请求过来时，nginx 根据 listen 和 server_name 这两个配置项来决定使用哪一个 server 块下的配置来处理请求。当请求头中的 “Host” 字段值匹配不到任一个 server_name 或者请求头中没有该字段时，nginx会把该请求路由到相关端口的默认 server 块中去处理。其中默认 server 是第一个定义在配置文件中的 server，也可以在 listen 配置项中明确指明：listen 80 default_server;

server_name的语法

此配置可以用 “精确名称、通配符、正则表达式” 来定义。

如果某个请求能匹配到多个 server_name，那么将按照如下的顺序，采用第一个匹配到的server来处理此请求：

  1. 精确名称
  2. 最长的以*开头的通配符定义，如”*.example.org”
  3. 最长的以*结尾的通配符定义，如”mail.*“
  4. 第一个匹配到的正则表达式（定义在配置文件中的顺序）

通配符定义

通配符定义包含一个 “星号*“ 并且 “星号*“ 只能出现在开头或结尾，并以 . 结尾或开头。下面两种格式 “www.*.example.org” 和 “w*.example.org”都是无效的，但可以用正则表达式来定义，比如，”~^www\..+\.example\.org\$” 和 “~^w.*\.example\.org\$”。“星号*” 可以匹配部分server name，”*.example.org”不仅匹配 www.example.org 而且也匹配 www.sub.example.org。

还有一种特殊的通配符定义”.example.org”，既可以匹配精确名称 “example.org” 又可以匹配通配符名称 “*.example.org”

正则表达式定义

nginx使用的是pcre正则，开头必须用 “符号~” 来定义，否则就会被当做是一个精确名称 或 通配符名称。^ 和 \$ 符号虽然不是语法上必须的，但是最好加上。域名中的 . 必须用 \ 转义，包含”{“和”}”的定义，必须用双引号外括：

PCRE库支持以下语法的正则命名捕获:

?\<name> Perl 5.10 compatible syntax, supported since PCRE-7.0
?’name’ Perl 5.10 compatible syntax, supported since PCRE-7.0
?P\<name> Python compatible syntax, supported since PCRE-4.0

如果nginx报错：

pcre_compile() failed: unrecognized character after (?\< in …

这说明PCRE库太旧，应该用”?P\<name>“语法来代替。

捕获也可以用数字引用，但仅限于简单形式，因为数字引用容易被覆盖。

混杂命名

前面也说过，如果想匹配未带有“Host”字段的请求，那么使用空字符串“”。

如果server name 定义为“\$hostname”，主机名将被使用。

如果直接访问ip来发起请求，那么“Host”字段值就是该ip，可以使用ip作为server name来匹配请求。

仅使用server_name指令无法定义可以匹配所有请求的server，但是可以定义 default server，来处理未能匹配到server name的请求。

性能建议

优先使用精确名称，其次是通配符名称，最后是正则。
